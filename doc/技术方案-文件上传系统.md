## 一、技术方案概述

### 1.1 背景

国货美妆集团各业务部门需定期向数据中台上报经营/营销数据，当前人工上报存在格式错误频发、监控缺失、IT 介入成本高等痛点。需建设一套统一文件上传系统，实现业务用户自助上传、模板化校验、可观测监控的闭环。

### 1.2 目标

| 目标 | 衡量标准 |
| :--- | :--- |
| 业务用户自助上传成功率 ≥ 95% | 成功任务数 / 总任务数 |
| IT 介入次数降为 0（除模板变更） | 人工处理工单数 |
| 2GB 文件上传完成时间 ≤ 10 分钟 | 分片上传 + 异步处理 |
| 历史列表查询响应 ≤ 2 秒（百万级记录） | 索引 + 分页优化 |

### 1.3 设计原则

1. **无状态服务**：所有节点无本地状态，支持水平扩展
2. **任务持久化**：异步队列 + 数据库双重保障，服务重启不丢任务
3. **最终一致性**：同步校验快速响应，异步完整校验通过飞书通知结果
4. **目录级权限隔离**：用户只能访问被显式授权的目录
5. **可观测优先**：每个关键节点埋点，支持 Grafana 监控

### 2.2 架构说明

- **前后端不分离**：前端使用 Thymeleaf 模板渲染，样式使用 Tailwind CSS 3.x，交互使用 Alpine.js，无独立前端工程。
- **分层架构**：严格遵循 Controller → Service → Mapper → DB 分层，Service 返回 DTO 而非实体类。
- **同步 + 异步双轨处理**：上传时同步执行表头/必填列快速校验（≤2s），校验通过后投递 MQ 消息，Worker 异步执行完整校验和数据写入。
- **文件存储双轨**：临时文件存储在服务器本地（或 NFS），处理成功后原始文件副本上传 OSS，保留 3 个月。
- **飞书深度集成**：登录使用飞书 OAuth2.0，通知使用飞书消息机器人 Webhook。

## 三、详细设计

### 3.1 核心流程

#### 3.1.1 飞书 OAuth2.0 登录时序

```
浏览器          Spring Boot          飞书开放平台
  │                 │                     │
  │──GET /login────►│                     │
  │◄──redirect──────│──302 redirect──────►│
  │                 │                     │ 用户扫码
  │◄────────────────────────callback──────│
  │──GET /callback?code=xxx────►│         │
  │                 │──POST /token───────►│
  │                 │◄──access_token──────│
  │                 │──GET /user_info────►│
  │                 │◄──open_id/name──────│
  │                 │ 写入/更新 sys_user   │
  │                 │ 写入 Session/JWT     │
  │◄──redirect /home─│                    │
```

**关键实现**：
- Session 存储：HttpSession 存储 `LoginUser`（userId、openId、roles）
- 首次登录自动创建账号，同步飞书部门信息
- 接口鉴权：`@PreAuthorize` + 自定义 `DirectoryPermissionAspect` 切面校验目录权限

#### 3.1.2 分片上传 + 断点续传时序

```
浏览器（JS）              Spring Boot
    │                        │
    │──POST /upload/init─────►│ 生成 taskId，返回 chunkSize=5MB, totalChunks
    │◄──{taskId, chunkSize}───│
    │                        │
    │──POST /upload/chunk─────►│ 上传第 n 片（含 taskId, chunkIndex, data）
    │   (循环 n=0..N)         │ 写入临时目录 /tmp/upload/{taskId}/chunk_{n}
    │◄──{chunkIndex: n}────────│
    │                        │
    │──POST /upload/merge─────►│ 所有分片完成，请求合并
    │                        │ 合并文件 → 同步快速校验（表头/必填列）
    │                        │ 校验失败 → 删除临时文件，返回错误
    │◄──{status, errors?}─────│ 校验通过 → 投递 MQ，任务状态=validating
    │                        │
    │──GET /tasks/{taskId}────►│ 前端轮询任务状态（可选）
```

**断点续传**：
- 前端每次上传前调用 `GET /upload/chunks?taskId=xxx` 查询已上传分片索引
- 服务端按 `taskId` 检查临时目录中已有分片，返回缺失分片列表
- 网络中断后重新上传时跳过已完成分片

#### 3.1.3 异步完整校验流程

```
MQ Consumer (ValidationWorker)
    │
    ├─ 1. 解析文件（POI/OpenCSV），流式读取（SXSSFWorkbook）
    ├─ 2. 表头字段名称匹配
    ├─ 3. 必填列检查
    ├─ 4. 字段类型校验
    ├─ 5. 枚举值校验
    ├─ 6. 正则表达式校验
    ├─ 7. 数值范围校验（可选）
    ├─ 8. 跨列依赖校验（可选）
    ├─ 9. 文件内唯一性约束（可选）
    ├─ 10. 与目标表唯一键冲突检查（可选，按策略处理：跳过/覆盖/整批拒绝）
    ├─ 11. 脏数据阈值判断（超阈值整批拒绝）
    │
    ├─ [校验失败] → upload_tasks.status=failed，写 upload_errors → 飞书通知失败
    │
    └─ [校验通过] → 批量写入 biz_* 业务表（JDBC批量插入）
                   → 原始文件上传 OSS
                   → upload_tasks.status=success → 飞书通知成功
```

**校验引擎设计**：责任链模式（Chain of Responsibility），每条规则实现 `ValidationRule` 接口，通过 `ValidationChain` 依序执行，支持短路（整批拒绝）和累积（收集所有行错误）两种模式。

```java
public interface ValidationRule {
    ValidationResult validate(ValidationContext context);
    int getOrder();
}
```

#### 3.1.4 监控看板定时任务

```
@Scheduled(cron = "0 0 * * * ?")  // 每小时执行
MonitorService.checkUploadStatus()
    │
    ├─ 查询所有配置了上传周期的最下级目录
    ├─ 判断当前时间是否在周期截止窗口内
    ├─ 统计各目录本周期上传状态（红/黄/绿）
    │
    ├─ 截止前 N 小时 → 发飞书提醒（业务用户 + 上级）
    └─ 超期未上传 → 更新监控状态为红，发飞书告警

@Scheduled(cron = "0 0 2 * * ?")  // 每日 02:00
CleanupService.cleanExpiredFiles()
    │
    ├─ 查询 upload_tasks.created_at < NOW() - 90天
    ├─ 删除本地临时文件 / OSS 文件
    └─ 软删除 upload_tasks / upload_errors 记录
```

### 3.3 接口设计

> 统一路径前缀：`/harness/api/v1`
> 统一响应格式：`{"code": 200, "msg": "success", "data": {}}`

#### 认证模块 `/auth`

| 方法 | 路径 | 说明 |
| :--- | :--- | :--- |
| GET | `/auth/feishu/login` | 重定向至飞书 OAuth 授权页 |
| GET | `/auth/feishu/callback` | 飞书回调，获取用户信息，写入 Session |

## 七、性能与容量设计

### 7.1 性能指标

| 指标 | 目标 | 实现方案 |
| :--- | :--- | :--- |
| 同步校验响应 | ≤ 2 秒 | 仅校验表头和必填列，不做完整解析 |
| 历史列表查询 | ≤ 2 秒（百万级） | 复合索引 (directory_id, status, create_time) + 分页 |
| 数据预览（前10行） | ≤ 1 秒 | 只读取文件前 N 行，使用 SAX 模式解析 |
| 2GB 文件上传 | ≤ 10 分钟 | 5MB 分片 + 并发 3 片，文件传输时间约 3-4 分钟 |
| 并发上传用户 | ≥ 50 | 无状态服务 + 连接池 20 + MQ Worker 异步处理 |

### 7.2 大文件解析优化

- **Excel**：使用 POI `SXSSFWorkbook`（流式读取，内存 ≤ 100 行缓存），避免 OOM
- **CSV**：使用 OpenCSV 流式读取，不一次性加载全文件
- **行数限制**：解析时计数超过 100 万行抛出 `UploadException`，终止解析

### 7.3 批量写入优化

```java
// MyBatis-Plus 批量插入，每批 1000 行，使用事务
@Transactional
public void batchInsert(String tableName, List<Map<String, Object>> rows) {
    Lists.partition(rows, 1000).forEach(batch ->
        dynamicTableMapper.batchInsert(tableName, batch)
    );
}
```

### 7.4 数据库索引规划


## 十、风险与应对

| 风险 | 等级 | 应对方案 |
| :--- | :--- | :--- |
| 飞书开放平台 API 不可用 | 高 | Session 缓存登录态（8h），告警降级为数据库记录 + 邮件 |
| 2GB Excel OOM | 高 | SXSSFWorkbook 流式解析，行数超 100 万前端拦截 |
| MySQL 写入高并发瓶颈 | 高 | 连接池 HikariCP + 每批 1000 行 JDBC 批量插入 + 异步队列限流（并发 Worker ≤ 10） |
| 分片文件丢失导致合并失败 | 中 | 分片记录入库，合并前校验所有分片存在；超时（1h）自动清理僵尸分片 |
| RabbitMQ 宕机任务丢失 | 中 | 队列持久化（durable=true）+ 消息持久化（deliveryMode=2）+ upload_tasks 表兜底补偿 |
| 动态 DDL 执行风险（SQL 注入） | 中 | 表名/列名严格白名单过滤（仅允许字母/数字/下划线），通过 JDBC 参数化执行 |
| 数据中台接入表结构不匹配 | 低 | 提前与数据中台对齐 biz_* 表命名规范和字段约束 |
